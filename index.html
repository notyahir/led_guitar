What It Does

A fretboard display that behaves like part of the instrument


FretGlow mounts a WS2812B 2.7mm 160 LEDs/m strip along the guitar fretboard and uses the DaisyPod as a hand controller. The project is meant to be readable from the player's point of view: pick a key, pick a scale, and the matching fret positions light up on the guitar.

The current prototype is intentionally simple. It focuses on getting the musical mapping, embedded controls, and LED signal chain working together reliably enough as a DIY project but also enough to demo in class.

Shows root notes and other scale tones with different colors so the tonal center stays visible.

Maps selected keys and scales onto string and fret positions using chromatic note offsets.

Uses the DaisyPod controls for key, scale, brightness, LED on/off state, and display mode.

Sends DaisyPod D10 data through an SN74AHCT125N level shifter before the LED strip.

Motivation

Making theory visible while playing


On guitar, a scale is not just a list of notes. It is a shape spread across strings, frets, hand position, and muscle memory. When practicing a new key, I wanted the information to live on the neck instead of on a chart next to the instrument.

The project came from that practical problem: can an embedded controller make the fretboard explain itself while I play? FretGlow is not meant to replace memorization or ear training but instead encourage it! 3

System Architecture

The signal path from DaisyPod to LEDs

The hardware chain is small but picky. DaisyPod D10 produces the LED data signal, the SN74AHCT125N shifts it from 3.3V to 5V logic, and the WS2812B strip receives one timing-sensitive data stream. The LEDs use a separate 5V supply, but all grounds are tied together.

Actual wiring notes

  • DaisyPod D10 connects to SN74AHCT125N pin 2.
  • SN74AHCT125N pin 3 drives the LED data line through a 470Ω resistor.
  • LED red wire connects to external +5V power.
  • LED white wire connects to ground.
  • DaisyPod ground, level shifter ground, and LED ground are shared.
  • A 1000µF capacitor sits across the LED power rails.
1

Control state

The DaisyPod reads the knobs, buttons, and encoder, then updates key, scale, brightness, and display mode in firmware.

2

Logic level

D10 data enters the SN74AHCT125N so the LED strip sees a cleaner 5V-compatible signal instead of the DaisyPod's 3.3V output.

3

Data protection

A 470Ω resistor sits in series with the data line near the LED input to calm the edge going into the first pixel.

4

Power

The external 5V supply powers the LEDs, with a 1000µF capacitor across 5V and GND and a shared ground for every board.

Hardware Build

A small circuit that had to be wired carefully

The circuit started on a breadboard so I could test quickly, but the LED strip made the weak points obvious. The final wiring worked better after shortening noisy connections, checking rails with a multimeter, and soldering fixes where the breadboard was not stable enough.

Build notes and parts

  • DaisyPod embedded controller
  • WS2812B 2.7mm 160 LEDs/m addressable RGB strip
  • SN74AHCT125N level shifter
  • External 5V power supply
  • Breadboard or prototyping board
  • 1000µF capacitor across 5V and GND
  • 470Ω resistor in series with the LED data line
  • Shared ground between DaisyPod, level shifter, and LEDs
  • 3D printed controller tray

Data line

DaisyPod D10 carries the WS2812B data signal. It goes into the SN74AHCT125N first, then through a 470Ω resistor before reaching the first LED's data input.

Power conditioning

The LED strip is powered by an external 5V supply. A 1000µF capacitor sits across 5V and GND to smooth the larger current changes that happen when many LEDs update.

Common ground

The DaisyPod, level shifter, LED strip, and external supply all share ground. When this reference was missing or loose, the strip produced unstable colors even when the firmware looked correct.

Firmware and Musical Mapping

Turning a selected scale into fret positions

The firmware is embedded C++ on the DaisyPod. It keeps a small musical state machine, maps strings and frets to pitch classes, and writes RGB values to the LED strip using WS2812B-compatible data timing.

Control mapping

One DaisyPod knob selects the key, another selects the scale, and the encoder or main knob adjusts brightness. One button toggles the LEDs on and off; another changes display mode.

Scales and modes

The firmware supports Major, natural minor, major pentatonic, minor pentatonic, blues, Dorian, and Mixolydian. Display modes are full scale, roots only, and all mapped frets test.

Fretboard mapping

Each string starts from its open note in standard tuning. The code maps 6 strings across 54 LEDs per row and uses measured fret-center positions for the first 12 frets.

LED timing

The WS2812B driver sends GRB bytes with direct GPIO writes and a DWT cycle counter. Timing presets made it possible to compare stable cyan-ish and magenta-ish behavior during debugging.

Fault handling

The physical strip has 640 LEDs, but the guitar map uses the first 324. Two calibrated bad pixels, indices 136 and 195, are forced off before every frame.

// Hardware path: DaisyPod D10 -> SN74AHCT125N -> 470R -> WS2812B DIN.
static constexpr Pin LED_DATA_PIN = D10;
const int BAD_LEDS[] = { 136, 195 };

// 160 LEDs/m = 6.25mm pitch, based on measured fret centers.
const int FRET_LED[FRETS] = {
  3, 8, 13, 17, 22, 26,
  29, 33, 36, 39, 42, 48
};

static void draw_scale_map()
{
  clear_leds();

  if(!strip_enabled)
    return;

  for(int s = 0; s < STRINGS; s++)
  {
    for(int f = 0; f < FRETS; f++)
    {
      int pc = pitch_class_for_position(s, f);
      int idx = led_index_for_position(s, f);

      if(mode == 2)
        set_rgb(idx, brightness, brightness, brightness);
      else if(pc == current_key)
        set_rgb(idx, brightness, 0, 0);
      else if(mode == 0 && in_scale(pc, current_key, current_scale))
        set_rgb(idx, 0, brightness / 3, brightness);
    }
  }
}

3D Printed Controller Tray

A red tray that made the prototype easier to handle

The tray holds the breadboard on top and the DaisyPod below so the controller does not become a loose pile of wires during testing. It also made it easier to bring the project to a desk, guitar stand, or demo setup without rebuilding the electronics every time.

The tray was designed around the exact prototype layout rather than an off-the-shelf enclosure. The open top was useful because the level shifter circuit still needed probing, rewiring, and soldered fixes late in the process.

A top cover would make the controller look more finished, but leaving it open made sense for this version because the project was still an engineering build, not a sealed product.

Debugging the LED Signal

Most of the trouble was electrical, not musical

The first versions did not fail politely. The LEDs flickered, showed colors I did not ask for, or stopped updating after a physical bump. That pushed the project into a debugging loop of measuring, rewiring, shortening leads, and isolating whether the problem was firmware, power, level shifting, or a bad LED.

Unstable WS2812B data

The WS2812B strip was sensitive to the exact data signal arriving at the first LED. When the D10 signal was weak, long, or noisy, the strip interpreted the timing incorrectly and produced flicker or wrong colors.

Noisy jumper wires

Some jumper connections looked fine until the guitar or tray moved. The breadboard made iteration fast, but it also introduced intermittent contact problems that were hard to distinguish from code bugs at first.

Multimeter checks

I used a multimeter to confirm 5V power, continuity, ground connections, and whether the SN74AHCT125N was actually receiving the expected supply and input wiring.

Soldered fixes

Cleaner physical connections helped more than cosmetic cleanup. Rewiring and soldering weak spots made the LED behavior noticeably more repeatable than the early all-jumper setup.

Tiny LED pads

The 2.7mm LEDs made repair work delicate. Too much heat or mechanical stress could damage a pixel or pad, so soldering fixes had to be small, quick, and carefully supported.

Faulty pixels

Two LEDs were unreliable enough that I disabled them in software. That was not the ideal fix, but it let the rest of the fretboard continue working for the final prototype.

Takeaway

The most useful debugging shift was treating the LED strip as an electrical signal-integrity problem instead of only a firmware problem. Once the level shifter, data resistor, capacitor, shared ground, and physical wiring were cleaned up, the same musical code became much easier to trust.

Final Result

A working prototype with clear next steps

By the end, FretGlow could display selected scales on the guitar neck, distinguish roots from other scale tones, and respond to DaisyPod input without reflashing. The build still has the feel of a student prototype, but it demonstrates the core interaction: changing a musical choice on the controller changes the instrument display immediately.

What worked

  • DaisyPod control of key, scale, brightness, and mode.
  • Root notes and scale tones shown in different colors.
  • Stable enough LED output after level shifting and cleaner wiring.
  • 3D printed tray holding the breadboard and DaisyPod as one unit.

What I would improve

  • Replace the breadboard with a custom PCB.
  • Use a more robust connector and strain relief for the LED strip.
  • Make the fretboard mounting cleaner and easier to remove.
  • Remove the need for software-disabled pixels by replacing damaged LED sections.
